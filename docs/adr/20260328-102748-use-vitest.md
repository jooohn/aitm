# Use Vitest as the test runner

**Date:** 2026-03-28
**Status:** accepted

## Context

The project needed a test runner for the Next.js backend. No testing infrastructure existed at the time.

## Decision

Use `vitest` with `vite-tsconfig-paths` for running tests. DB isolation in tests is achieved by setting the `AITM_DB_PATH` environment variable to `:memory:` via `vitest.config.ts`, causing `src/lib/db.ts` to open an in-memory SQLite database instead of the real `~/.aitm/aitm.db`.

## Consequences

- Vitest supports TypeScript natively without a separate Babel/ts-jest transform, and the `vite-tsconfig-paths` plugin resolves the `@/*` path alias automatically.
- Each test file runs in its own worker, giving it an isolated module registry and therefore an isolated in-memory database.
- The `AITM_DB_PATH=:memory:` isolation relies on the env var being applied before `db.ts` is first imported. This is guaranteed by vitest's worker setup, but is worth keeping in mind if the DB initialization is ever restructured.
- Jest was not chosen because it requires additional configuration (`ts-jest` or Babel) for TypeScript and the `@/*` alias.

## Alternatives considered

- **Jest** — more widely used in the Next.js ecosystem but requires extra TypeScript and path-alias setup.
- **Node.js built-in test runner (`node:test`)** — no additional dependency, but lacks the watch mode, snapshot, and coverage tooling that vitest provides out of the box.
